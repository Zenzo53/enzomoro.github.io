GitHub Actions automation for continuous security, quality, and performance

SAST: CodeQL for JavaScript

What & Why

CodeQL is GitHub's industry-leading static analysis engine that identifies security vulnerabilities and code quality issues at the source level. For JavaScript, it detects:

  • XSS (Cross-Site Scripting) vulnerabilities
  • SQL Injection risks
  • CSRF (Cross-Site Request Forgery) patterns
  • Insecure cryptography usage
  • Path traversal flaws

Where to See Results: Check the Security tab ‚Üí Code scanning in the repository settings. Alerts are prioritized by severity.

DAST: OWASP ZAP Baseline

What & Why

OWASP ZAP (Zed Attack Proxy) is a dynamic security scanner that tests the live application against OWASP Top 10 threats. It runs automated scans against the deployed GitHub Pages site on a schedule (nightly) and on manual trigger.

Baseline Mode performs rapid, low-risk passive scanning suitable for CI/CD pipelines. Detects:

  • Missing security headers (CSP, X-Frame-Options, etc.)
  • Cookie misconfigurations
  • SSL/TLS issues
  • Information disclosure leaks
  • Outdated library versions

Workflow: Runs on .github/workflows/zap-baseline.yml. High/Medium alerts cause workflow failure; Low severity are logged.

Secret Scanning: Gitleaks

What & Why

Gitleaks detects sensitive information (API keys, passwords, tokens, private keys) accidentally committed to version control. It runs on every push and pull request to prevent secrets from reaching the repository or being exposed in history.

Policy: Any detected secret causes the workflow to fail and block the commit. This enforces a zero-tolerance approach for credential exposure.

Best Practices:

  • Use environment variables or GitHub Secrets for sensitive data
  • Never hardcode credentials in code
  • Use `.gitignore` for `.env` and config files
  • Rotate any exposed credentials immediately

Link Checker: Lychee

What & Why

Lychee scans the entire site for broken links, ensuring all internal and external references are valid. Broken links harm user experience and SEO.

Workflow: Runs on every push. Fails on dead links; ignores error pages (404) by configuration.

Content Security Policy (CSP)

What & Why

CSP mitigates XSS, clickjacking, and injection attacks by restricting the sources from which content can be loaded. This site uses a Report-Only policy to monitor violations without blocking functionality.

Recommended CSP (Report-Only): 

<meta http-equiv="Content-Security-Policy" content="
  default-src 'self';
  script-src 'self' 'unsafe-inline' https://plausible.io;
  style-src 'self' 'unsafe-inline' https://fonts.googleapis.com;
  img-src 'self' data: https:;
  font-src 'self' https://fonts.gstatic.com;
  connect-src 'self' https://formspree.io https://plausible.io;
  frame-ancestors 'none'
" />

GitHub Pages Limitations: Pages doesn't support HTTP headers for CSP, so the meta tag is the primary enforcement mechanism. See SECURITY.md for full CSP guidance.

Software Bill of Materials (SBOM)

What & Why

A Software Bill of Materials (SBOM) is an inventory of all software components and dependencies. This static site has minimal build dependencies; however, third-party assets are documented:

Third-Party Assets

  • Fonts: System UI font stack (no external CDN)
  • Icons: Unicode symbols (no dependencies)
  • Analytics (Optional): Plausible Analytics (privacy-focused)
  • Forms (Optional): Formspree (for serverless form handling)

No npm/build dependencies are committed. All security tooling runs in GitHub Actions CI/CD.
